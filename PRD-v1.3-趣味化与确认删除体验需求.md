> **日期**: 2026-05-31

## 1. 版本目标

v1.3 的目标是在不改变核心照片整理流程的前提下，提升关键决策点的质感，并为产品加入轻量趣味化能力，让用户在完成清理后获得更明确的反馈、更强的成就感和持续使用动机。

本版本聚焦 4 个方向：

1. 重设计确认删除对话框，使删除前确认更符合当前暗色、金色强调的整体风格。
2. 强化个人中心统计卡片，让连续天数、最近删除、释放空间等核心数据更醒目。
3. 在个人中心新增“每周清理回顾”功能。
4. 在个人中心新增“成就系统”功能。

---

## 2. 设计参考

### 2.1 iOS / Apple HIG 参考原则

参考 Apple Human Interface Guidelines 中关于 Alerts 的建议：

- 删除确认属于破坏性操作，应只在确有必要时打断用户。
- 文案应短、明确、可行动。
- 操作按钮需要清晰表达后果，避免只写“确定”这类模糊文案。
- 删除类操作应保留取消路径，并降低误触风险。

参考链接：

- Apple HIG - Alerts: https://developer.apple.com/design/human-interface-guidelines/alerts/
- Apple UIKit Alerts and Action Sheets: https://developer.apple.com/documentation/uikit/windows_and_screens/getting_the_user_s_attention_with_alerts_and_action_sheets

### 2.2 视觉灵感参考

参考 Dribbble 上 confirmation modal、achievement mobile UI 的常见趋势：

- 半透明深色遮罩 + 居中卡片或底部 sheet。
- 大数字/大图标作为视觉锚点。
- 危险操作使用红色，但不大面积铺满，避免廉价感。
- 成就系统常使用徽章、进度环、解锁态/未解锁态对比。

参考链接：

- Dribbble Confirm Action Modal: https://dribbble.com/search/confirm-action-modal
- Dribbble Achievements UI: https://dribbble.com/tags/achievements-ui

## REQ-01: 确认删除对话框重设计

### 背景

当前 `DeleteConfirmSheet` 已经是自定义弹窗，但视觉仍偏基础：主要由标题、说明文案、删除按钮、取消按钮组成。用户希望参考 iOS design 和优秀移动端 UI 案例，将确认删除界面做得更精致，并符合 PickUp 当前暗色 + 金色强调的风格。

### 目标

将确认删除对话框升级为更有质感、更清晰、更符合破坏性操作心智的自定义弹窗。

### 设计方向

推荐方案：**iOS 风格玻璃拟态危险确认卡片**。

核心视觉：

- 深色遮罩，背景轻微压暗。
- 中央浮层卡片，圆角 28 左右。
- 顶部使用危险操作图标，例如 `delete-outline` 或 `alert-circle-outline`。
- 主视觉突出删除数量，例如：
  - `删除 12 张照片？`
  - 数字 `12` 使用更大字号。
- 文案说明保持短句：
  - `这些照片会进入系统最近删除，你仍可在限定时间内恢复。`
- 主按钮使用红色危险色。
- 取消按钮使用低强调透明按钮。
- 可选展示一条轻量提示：
  - `删除前已为最近删除记录生成本地缩略图。`

### 交互规则

| 场景 | 行为 |
|---|---|
| 点击删除按钮 | 执行 `onConfirm` |
| 删除中 | 删除按钮进入 loading 态，禁用重复点击 |
| 点击取消 | 执行 `onCancel` 并关闭 |
| 点击遮罩 | 关闭弹窗 |
| Android 返回键 | 关闭弹窗 |
| count = 1 | 文案使用单张语义 |
| count > 1 | 文案使用批量语义 |

### 动效要求

- 弹窗出现：遮罩 180-240ms fade in，卡片 scale 0.96 -> 1，opacity 0 -> 1。
- 弹窗关闭：遮罩 fade out，卡片轻微 scale down。
- 删除按钮点击时可加入轻微 press scale，不超过 0.97。
- 动效要克制，不做夸张弹跳。

### 技术范围

主要修改：

- `src/components/delete-review/DeleteConfirmSheet.tsx`

可选修改：

- `src/design-tokens.ts`
- `src/components/ui/SafeBlurView.tsx`

### 验收标准

- [ ] 删除确认弹窗不使用原生 `Alert.alert`。
- [ ] 弹窗视觉与当前黑色背景、黄色强调风格一致。
- [ ] 主删除动作是明确的危险按钮，取消动作清晰可见。
- [ ] 删除中不能重复点击确认。
- [ ] Android 返回键可关闭弹窗。
- [ ] iOS / Android 布局不溢出、不遮挡、不出现文字截断。
- [ ] `npx.cmd tsc --noEmit` 通过。
- [ ] `npx.cmd jest --runInBand` 通过。

## REQ-02: 个人中心统计卡片醒目色彩强化

### 背景

当前个人中心统计卡片视觉偏统一，核心数据不够醒目。用户希望连续天数、最近删除等区域的字体颜色变成黄色、绿色等更醒目的颜色，让统计区更有活力。

### 目标

在保持整体暗色 UI 统一的前提下，为不同统计指标分配语义化强调色，提高信息识别度。

### 色彩建议

| 卡片 | 建议强调色 | 语义 |
---|---|---|
| 已浏览 | `#FFFFFF` 或浅灰 | 基础浏览数据，低情绪 |
| 最近删除 | `#FF453A` 红色 | 删除/清理动作 |
| 连续天数 | `#34C759` 绿色 | 连续、成长、正反馈 |
| 释放空间 | `#FFCC00` 黄色 | 空间收益、成就感 |

### UI 规则

- 只强化数值，不建议整张卡片大面积变色。
- label 保持较低强调，避免统计区过于嘈杂。
- 数字颜色可通过 `StatCard` 新增 `valueColor` 参数实现。
- 可为卡片添加非常轻的同色描边或背景 tint，但透明度需低。
- 不使用彩虹式混乱配色，保持 3-4 个明确语义色。

### 技术范围

主要修改：

- `src/components/settings/StatCard.tsx`
- `app/settings.tsx`

### 验收标准

- [ ] 最近删除数值为红色系。
- [ ] 连续天数数值为绿色系。
- [ ] 释放空间数值为黄色系。
- [ ] 已浏览保持白色或浅灰，避免喧宾夺主。
- [ ] 颜色在暗色背景下具备足够对比度。
- [ ] 不影响 StatCard 原有布局。

## REQ-03: 每周清理回顾

### 背景

用户完成照片整理后，目前只能看到静态统计数据。缺少周期性反馈，无法形成“这一周我整理了多少”的成就感。每周清理回顾可以把已有数据转化为更有趣、更可感知的总结。

### 目标

在个人中心中部新增“每周清理回顾”模块，让用户看到最近 7 天的清理成果。

### 功能入口

位置：个人中心中部，建议位于“统计”区下方、“帮助”区上方。

模块标题：

- `每周清理回顾`

展示形态：

- 一张横向回顾卡片。
- 不需要进入二级页面，v1.3 先做单卡片展示。

### 展示内容

每周回顾卡片展示：

| 字段 | 数据来源 | 示例 |
|---|---|---|
| 本周浏览 | `weeklyHistory.viewed` 求和 | `本周浏览 126 张` |
| 本周删除 | `weeklyHistory.deleted` 求和 | `清理 18 张` |
| 连续天数 | `streakDays` | `连续整理 4 天` |
| 本周结论文案 | 派生规则 | `这周你给相册松了一口气` |

可选视觉：

- 一条 7 天 mini bar。
- 每天一个小柱，显示当天删除数量。
- 当前没有数据时显示轻空态：
  - `本周还没有清理记录`
  - `整理一组照片后，这里会生成你的周回顾`

### 文案规则

| 条件 | 文案 |
|---|---|
| deleted = 0 | `本周还没有清理记录` |
| deleted 1-9 | `这周完成了一次轻整理` |
| deleted 10-49 | `这周相册清爽了不少` |
| deleted >= 50 | `这周你完成了一次大扫除` |

### 数据规则

- 使用已有 `weeklyHistory`，不新增云端数据。
- 按最近 7 条或最近 7 天数据展示。
- 如果当天无记录，对应 bar 显示为 0。
- 该模块只读，不修改统计数据。

### 技术范围

可能新增：

- `src/components/settings/WeeklyReviewCard.tsx`

可能修改：

- `app/settings.tsx`
- `src/contexts/StatsContext.tsx`（如当前未暴露 `weeklyHistory`，需要补充暴露）
- `src/services/stats-service.ts`

### 验收标准

- [ ] 个人中心展示“每周清理回顾”模块。
- [ ] 有数据时展示本周浏览、本周删除、连续天数和总结文案。
- [ ] 无数据时展示空态。
- [ ] 7 天 mini bar 数据与 `weeklyHistory` 一致。
- [ ] 不影响现有统计卡片。
- [ ] App 重启后数据仍正确。

## REQ-04: 成就系统

### 背景

照片清理本质上是低频、带负担的工具型行为。成就系统可以把整理过程转化为轻量游戏化反馈，提高完成感和留存。

### 目标

在个人中心中部新增“成就系统”模块，展示用户已解锁/未解锁的整理成就。

### 功能入口

位置：个人中心中部，建议与“每周清理回顾”相邻。

模块标题：

- `成就`

展示形态：

- v1.3 先做横向滚动徽章列表。
- 每个徽章包含图标、名称、状态、进度。
- 不做复杂二级页，减少范围。

### 首批成就设计

| 成就 | 解锁条件 | 图标建议 | 色彩 |
|---|---:|---|---|
| 初次整理 | `totalViewed >= 1` | `sparkles` | 黄色 |
| 清理新手 | `recentDeleteCount >= 10` 或 `totalDeleted >= 10` | `delete-sweep-outline` | 红色 |
| 空间救星 | `totalFreedBytes >= 500MB` | `database-arrow-down-outline` | 黄色 |
| 连续行动 | `streakDays >= 3` | `fire` / `calendar-check` | 绿色 |
| 相册管家 | `totalViewed >= 500` | `image-multiple-outline` | 蓝色 |

### 进度规则

- 已解锁：高亮显示，展示完成态。
- 未解锁：低透明度显示，展示 `当前 / 目标`。
- 进度不可超过 100%。
- 成就数据本地派生，不需要单独持久化。
- 如后续需要“首次解锁弹窗”，另开需求，不放入 v1.3。

### 推荐展示文案

| 状态 | 示例 |
|---|---|
| 已解锁 | `已解锁` |
| 未解锁 | `7 / 10` |
| 空态 | `开始整理照片后会点亮成就` |

### 技术范围

可能新增：

- `src/components/settings/AchievementStrip.tsx`
- `src/utils/achievement-utils.ts`

可能修改：

- `app/settings.tsx`
- `src/contexts/StatsContext.tsx`

### 验收标准

- [ ] 个人中心展示“成就”模块。
- [ ] 至少包含 5 个首批成就。
- [ ] 成就状态根据本地统计数据实时计算。
- [ ] 未解锁成就展示进度。
- [ ] 已解锁成就视觉高亮。
- [ ] 不引入云端依赖。
- [ ] App 重启后成就状态保持一致。

## 10. 风险与注意事项

- `StatsContext` 当前如果没有暴露 `weeklyHistory`，需要补充上下文数据出口。

## REQ-05: 待删除确认页布局上移与视觉重心调整

### 背景

当前待删除确认页的左上角返回按钮位置偏低，页面中部“准备收工了吗”文案与下方照片堆叠区域也偏下。整体视觉重心落在屏幕下半部分，导致页面看起来不够紧凑，尤其在小屏或高屏设备上，顶部留白与主体内容关系不自然。

### 目标

优化待删除确认页的垂直布局，使返回按钮、标题区和照片堆叠区整体上移，形成更稳定、更精致的确认页构图。

### 设计规格

| 区域 | 当前问题 | 目标调整 |
|---|---|---|
| 左上角返回按钮 | 距离顶部偏大，视觉上像落在内容区内 | 上移约 `8-16px`，贴近安全区下方 |
| 主标题区 | “准备收工了吗”偏低，页面顶部留白过多 | 标题与副标题整体上移约 `12-24px` |
| 照片堆叠区 | 与标题距离偏松，且整体偏下 | 随标题一起上移，保持标题与图片之间的自然间距 |
| 底部操作区 | 不应被上移影响 | 保持 Delete / 稍后删除按钮安全区适配 |

### 交互规则

- 返回按钮点击行为保持不变。
- “Delete”主按钮、“稍后删除”次按钮行为保持不变。
- 页面内容上移不能导致小屏设备上照片堆叠与按钮重叠。

## REQ-06: Delete 按钮文件大小 0B 修复

### 背景

待删除确认页的 `Delete` 按钮后方会展示本次预计释放空间，但当前一直显示 `0B`。这通常说明待删除照片的 `fileSize` 没有正确传入确认页，或删除前文件大小获取逻辑对 `content://` / `ph://` 等媒体 URI 不兼容，导致累计值始终为 0。

### 目标

修复本次待删除照片空间统计链路，让 `Delete` 按钮展示可信的预计释放空间，例如：

- `Delete · 8.4 MB`
- `Delete · 126 MB`

当无法获取真实文件大小时，应使用稳定估算值兜底，不允许在有照片待删除时展示 `0B`。

### 数据规则

| 场景 | 目标行为 |
|---|---|
| 照片已有 `fileSize > 0` | 直接使用 `fileSize` |
| `fileSize = 0` 或缺失 | 使用文件系统 API 尝试获取真实大小 |
| 文件系统 API 失败 | 使用像素估算法兜底 |
| 多张照片 | 对每张照片大小求和 |
| 视频 / Live Photo | 若无法准确读取，优先使用可取得的媒体文件大小；否则用像素估算并避免显示 0 |

### 估算策略

当真实文件大小不可用时，使用当前项目已采用的像素估算法：

```ts
Math.round(width * height * 3 * 0.4)
```

该值作为 JPEG 压缩后的近似估算，不追求绝对精确，但必须避免用户看到 `0B`。

### 技术范围

主要修改：

- `src/services/delete-service.ts`
- `src/utils/delete-confirm-utils.ts`
- `src/components/delete-review/DeleteConfirmSheet.tsx`

可能涉及：

- `src/types/photo.ts`
- `app/review.tsx`
- `app/index.tsx`

### 验收标准

- [ ] 有待删除照片时，`Delete` 按钮后方不再显示 `0B`。
- [ ] 多张照片时展示总大小，而不是单张大小。
- [ ] 文件大小格式化正确：`B / KB / MB / GB`。
- [ ] Android 媒体 URI 无法直接读取时，仍能通过估算值展示非 0 数字。
- [ ] 删除成功后，个人中心“释放空间”累计值与本次删除结果保持同一套计算逻辑。
- [ ] 单元测试覆盖 `fileSize = 0` 时的兜底估算。

## REQ-07: Review 照片双指捏合与双指点击缩放

### 背景

Review 页面用于最终确认待删除照片，是避免误删的关键场景。当前虽然已有照片预览能力，但双指捏合和双指点击照片仍无法可靠放大，用户不能快速查看照片细节。

### 目标

在 Review 照片预览中实现可靠、自然的缩放体验：

1. 双指捏合可以连续放大 / 缩小。
2. 双指点击可以在放大和还原之间切换。
3. 放大后可以拖动查看局部细节。

### 交互规格

| 手势 | 行为 |
|---|---|
| 双指捏合向外 | 照片放大，最大 `5x` |
| 双指捏合向内 | 照片缩小，最小回弹到 `1x` |
| 双指点击照片 | 若当前为 `1x`，放大到 `2x`；若当前大于 `1x`，还原到 `1x` |
| 放大后单指拖动 | 平移照片，查看局部 |
| 关闭预览后再次打开 | 缩放和平移状态重置 |
| 单击关闭区域 / 关闭按钮 | 关闭预览，回到 Review 网格 |

### 设计边界

- 缩放范围建议：`1x ~ 5x`。
- 双指点击是辅助快捷操作，不能影响单击关闭或长按预览逻辑。
- 手势冲突优先级：缩放预览内的手势优先于页面滚动。
- 关闭预览后，Review 网格中已勾选 / 未勾选状态不能丢失。

### 技术方案

优先复用并修复现有：

- `src/components/delete-review/PhotoZoomModal.tsx`

建议实现方式：

- 使用 `react-native-gesture-handler` 的 Gesture API 或现有项目兼容的手势组件。
- 使用 `react-native-reanimated` 管理 `scale`、`translateX`、`translateY`。
- 在预览打开时重置 shared values。
- 对平移范围做边界限制，避免照片被拖出屏幕后无法找回。

### 技术范围

主要修改：

- `src/components/delete-review/PhotoZoomModal.tsx`

可能涉及：

- `src/components/delete-review/DeleteGrid.tsx`
- `app/review.tsx`

### 验收标准

- [ ] Review 页面进入照片预览后，双指捏合可以平滑放大和缩小。
- [ ] 双指点击照片可以在 `1x` 与约 `2x` 之间切换。
- [ ] 放大后可拖动查看照片局部。
- [ ] 关闭预览再打开另一张照片时，缩放状态重置为 `1x`。
- [ ] 缩小低于 `1x` 时自动回弹，不出现照片过小或漂移。
- [ ] 缩放手势不影响 Review 网格的勾选状态。
